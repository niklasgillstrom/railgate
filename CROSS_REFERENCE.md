# Cross-reference — Article claims ↔ reference-implementation code

**Purpose.** This table maps every testable technical claim made in Article 1 (Gillström, in preparation; target venue: *Capital Markets Law Journal*), Article 2 (Gillström, in preparation; target venue: *Computer Law & Security Review*), and the `HARDWARE_BASELINE.md` document to a concrete code path, test, documentary artefact, or an honest "gap" flag. It is intended to let a peer reviewer confirm, claim-by-claim, whether an assertion is backed by executable code, by documentary evidence, by the article's own argument, or by nothing yet.

**Status codes.**

- **CODE** — the claim is backed by executable code in one of the two repos under this tree. Reviewer can exercise the path with `mvn -B test` or by running the application.
- **DOC** — the claim is backed by `HARDWARE_BASELINE.md`, by an external vendor document, or by a published certification/standard whose URL is cited.
- **TEXT-ONLY** — the claim is theoretical, normative, or citation-based. It does not require code backing (e.g. a reference to Julia Black 2008, a DSR methodology citation, a reading of DORA recitals). Listed here so reviewers can see that the claim has been considered and classified.
- **GAP** — the claim plausibly *should* have executable or documentary backing in these repos and does not yet. Flagged so that a reviewer can see honest limitations up front.

**Repository references.** Paths are rooted at `hsm/` (shorthand: `[HSM]`), `gatekeeper/` (shorthand: `[GK]`), or `railgate/` (shorthand: `[RG]`). File paths are given relative to these roots. **This document is shipped identically across the three repositories of the triadic system** (hsm + gatekeeper + railgate); whichever repo a reviewer is reading from, the same `CROSS_REFERENCE.md` lives in the sibling repos and resolves the cross-repo paths to executable code there.

**Triadic system overview (since v1.2.0).** The reference implementations form a three-component system:

- **`[HSM]`** — financial-entity-side HSM attestation verification core ([10.5281/zenodo.19930310](https://doi.org/10.5281/zenodo.19930310), concept DOI)
- **`[GK]`** — NCA-operated supervisory gatekeeper API; from v1.1.0 also exposes `POST /api/v1/verify` for settlement-time signature verification ([10.5281/zenodo.19930395](https://doi.org/10.5281/zenodo.19930395), concept DOI)
- **`[RG]`** — central-bank settlement-rail enforcement layer; calls `[GK]/api/v1/verify` at settlement time ([10.5281/zenodo.19952991](https://doi.org/10.5281/zenodo.19952991), concept DOI)

The three components together operationalise the data-minimised quadruple-triangulation model described in Article 1 §4.2 and Article 2 §9.3.

**External-document references.** Citations to `HARDWARE_BASELINE.md §X.Y` resolve to a separate document — *Gillström, HARDWARE_BASELINE.md (in academic submission package)* — that is **not** part of either code repository. It is a sibling artefact to Article 1 (CMLJ) and Article 2 (CLSR) in the academic submission folder, documenting the operational case-study deployment (Securosys Primus, YubiHSM, FIPS-mode configuration, audit-log pipeline, time-sync infrastructure, attack-vector review). Peer reviewers receive HARDWARE_BASELINE.md as part of the submission bundle alongside the two articles and these two repositories; it is intentionally excluded from the public MIT-licensed repos because it carries device serial numbers, invoice references, and operational configuration that belong in a controlled submission folder, not in a public Zenodo deposit. Anonymous Zenodo readers will see the citations but cannot resolve them; that is the intended scope.

## GAP summary

The following items remain flagged GAP after the implementation work documented in this repository. A peer reviewer should interpret these as honest limitations of the reference implementation at submission time — items that the articles describe but the code does not yet demonstrate executable backing for. The GAPs that earlier drafts of this document listed and that have since been closed (persistent tamper-evident registry, rate limiting, NCA-role authorisation, production receipt-signing key, signed batch export) are no longer listed here; the rows above carry their CODE-status entries directly.

1. **Triangulation across three independent data sources** (Art 1 §4.2, §5.2, cross-cutting) — only the vendor-root branch is implemented end-to-end. The other two ingestion streams (banks-report, GetSwish-report) are described in the article and operationalised by the supervisor's manual data triangulation procedure documented in `[GK]/SUPERVISORY_OPERATIONS.md` §3.5, but they are not joined as a single in-code check.
2. **Organisational (ISMS-internal) gatekeeper deployment** (Art 2 §Abstract, §9.3) — the repo ships the supervisory variant; the organisational variant is architecturally compatible but not separately packaged.
3. **Schneier-Kelsey-style hash-chained forward-secure event log with COSE encoding and RFC 3161 timestamping** (Art 2 §5.1 FR5, §6.3, §8.5) — partially closed: per-receipt signing exists (`ConfiguredReceiptSigner`), the audit log is hash-chained and per-entry-signed (`AppendOnlyFileAuditLog`), and the approval registry has its own append-only journal (`AppendOnlyFileApprovalRegistry`). What remains: COSE encoding instead of JSONL, RFC 3161 timestamping integration with a TSA, and forward-secure key rotation per Ma–Tsudik (2008). JSONL is intentionally retained over CBOR/COSE for peer-review reviewability; the article identifies the COSE migration as future work.
4. **Qualified TSA integration** (Art 2 §5.3 STR5, §8.5; cross-cutting) — the underlying hardware (Securosys Primus) has RFC 3161 capability *licensed and activated* (per `HARDWARE_BASELINE.md` §3.1 "Device capabilities available but not used"); the gatekeeper `ReceiptSigner` does not yet integrate a qualified TSA. No qualified-TSP wiring ships with the reference.
5. **Full signing-key lifecycle events** (Art 2 §6.2 — Generated / Issued / Signed / Rotated / Revoked / Decommissioned) — only a subset (verify and confirm) is captured by `ApprovalRegistry`. Rotated/revoked/decommissioned event types are not yet emitted.
6. **Marvell TLV attestation-blob parser for Azure / Google** — speculative; blob-layout assumptions not verified against Marvell's published specification. Flagged in the SECURITY NOTE block above the `ATTESTATION_TRUST_ANCHOR` constant in each cloud-HSM verifier.
7. **Dual-chain owner-root validation for cloud HSMs** — Google's published verification model (`verify_chains.py`, 2021) anchors at both the Marvell manufacturer root and Google's "Hawksbill Root v1 prod" owner root; Azure is expected to follow an analogous pattern with a Microsoft-controlled owner root. Only the manufacturer chain is implemented.
8. **Real signatory-rights registry adapter** (Swish agreement registry / Bolagsverket) — `[HSM]` ships only `FailClosedSignatoryRightsVerifier` (production-safe default; rejects every query) and `MockAgreementRegistrySignatoryRightsVerifier` (JSON file). The FE must implement a custom adapter against its authoritative registry.

These GAPs do not invalidate the articles' central claims, which are *demonstrated* by the CODE rows above. They do delimit the scope of the reference implementation versus the scope of the argument: the articles argue what the mechanism does in principle and what the case-study hardware instantiates in practice, while the reference implementation demonstrates the verifier core, the supervisory gatekeeper shell, and the FE-side integration pattern. Closing the GAPs is a production-deployment concern, not a scientific-claim concern, and each is documented either inline in this table or in `HARDWARE_BASELINE.md`.
